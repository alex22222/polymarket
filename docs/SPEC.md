# Polymtrade Spec

> Spec 模式目标：让系统围绕同一个长期目标做循环工程，而不是被短期样本、单次盈亏或临时灵感带偏。

---

## 1. 最终目标

用最小成本证明或证伪这个核心假设：

> Polymtrade 的模型概率在特定 BTC/ETH barrier 市场上，长期比市场隐含概率更接近真实触及概率，并且扣除费用、滑点、盘口限制后仍有正期望。

系统不是为了“每天找到交易”，而是为了回答：

- 哪类市场存在真实 edge？
- 模型在哪些情形会系统性高估或低估？
- 数据质量是否足够支撑决策？
- 机会在盘口里是否真实可执行？
- 小额真实测试是否有统计依据？

---

## 2. 非目标

以下事项默认不做，除非有明确人工批准：

- 自动真实下单或自动平仓。
- 为了看到候选而降低交易纪律。
- 用少量样本直接调参。
- 用截图或手工输入替代可验证数据源作为核心数据。
- 把短期 PnL 当成模型有效性的证据。
- 扩大币种、市场类型或频率来制造样本量，而不先评估数据质量和执行成本。

---

## 3. 阶段拆解

### Phase 0：数据可信

目标：确保价格、K 线、市场元数据、盘口、宏观事件、自动任务都稳定。

验收条件：

- 自动化任务连续运行，无阻断级错误。
- BTC/ETH 实时价格可用。
- K 线新鲜度在 24 小时内。
- Scanner 使用实时盘口时能识别过期盘口、宽 spread、部分成交。
- 数据可信状态在页面和日报中可见。

### Phase 1：样本积累

目标：积累足够的候选、准候选、研究机会和 paper trading 样本。

验收条件：

- 候选复盘已结算样本达到 50 条：允许做初步判断。
- 候选复盘已结算样本达到 100 条：允许做粗分组判断。
- Paper trading 已结算样本达到 30 条：允许评估严格交易规则的初步表现。
- Paper trading 已结算样本达到 50 条：允许讨论小额真实测试。

### Phase 2：模型校准

目标：判断模型概率是否比市场概率更准。

验收条件：

- 模型 Brier 与市场 Brier 可比较。
- 概率桶有足够 resolved 样本。
- 能识别模型高估/低估的方向、资产、到期、盘口质量、波动率来源。

### Phase 3：策略收敛

目标：把有效模式收敛成严格、可执行、低频的小额策略。

验收条件：

- 正 ROI 不是由单个样本驱动。
- 分组表现跨时间稳定。
- 盘口可成交、滑点和费用后仍有净 edge。
- 有明确止盈、止损、退出和最大敞口限制。

### Phase 4：小额真实测试

目标：只在样本和执行质量达标后，用极小金额验证真实成交与心理/流程成本。

验收条件：

- Paper trading 表现达标。
- TODO 中无 P0 数据可信问题。
- 单笔真实测试金额可完全亏损。
- 每笔测试必须能复盘：入场理由、盘口、模型概率、市场价格、退出策略、结果。

---

## 4. 决策门槛

| 决策 | 最低门槛 |
|---|---|
| 调整模型核心参数 | 已结算候选 >= 50，且能说明偏差来自模型而不是数据 |
| 调整交易阈值 | Paper trading 已结算 >= 30，且风险后 ROI 有改善 |
| 扩大币种范围 | BTC/ETH 数据链路稳定，且新增币种有可靠现货、K 线、盘口数据 |
| 上线真实小额交易 | Paper trading 已结算 >= 50，且无 P0 数据可信问题 |
| 自动交易 | 暂不允许；需要单独 spec 和人工批准 |

---

## 5. 重要改进前检查

任何重要模型、策略、执行、自动化改动前，都要先回答：

1. 这个改动服务最终目标吗？
2. 它改善的是数据、模型、执行，还是只是让候选变多？
3. 它是否可能引入过拟合？
4. 它是否降低了风控或交易纪律？
5. 它是否依赖未验证数据源？
6. 它会不会让 paper trading 和真实执行口径不一致？
7. 如果改动失败，如何发现和回滚？

建议运行：

```bash
python3 -m polymtrade.research.spec_guard --proposal "描述这次重要改动"
```

## 6. 当前经验

手工维护区：记录经过人工确认的长期经验。

- 系统的第一目标是证伪/证明 edge，不是每天交易。
- 已结算样本数比总观测数重要。
- 盘口可执行性必须和模型概率同等重要。
- 最后几小时高胜率市场通常 ROI 很薄，不能只看胜率。
- 二元合约买错时本金可归零，所有 ROI 必须和最大亏损一起展示。
